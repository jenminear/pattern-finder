# Agent Spec — Pattern Finder

## Overview
A system that recognizes patterns in small, labeled, tabular scenarios (up to 5 input fields, each user-labeled), stores discovered patterns for reuse, and applies them to guess the "consequence" of new scenarios. Built for a Kaggle AI Agents capstone (Freestyle category), demonstrating three required concepts — Agent Skills, MCP for database access, deployment on Google Cloud — plus a fourth, additional concept: A2UI for the agent's own output surfaces.

The agent is invoked at two distinct handoff points from a custom web UI (not a general chat interface):

1. **Guess Phase** — triggered when the user clicks "Guess the Consequence" after entering a scenario, but only after a cheap deterministic DB lookup (exact label-set + existing pattern match) comes up empty. The agent searches for a pattern among same-labeled scenarios, falls back to label-independent pattern matching, and returns either a consequence estimate or "I don't know" based on a single user-adjustable **effort dial** (a UI slider, session-only, not persisted to the DB — see Effort Dial section below).
2. **Learn Phase** — triggered after the user submits the correct consequence. The agent compares its guess to the actual answer and, on a match, captures/updates the pattern (new pattern record with abstracted text description + rule/code, or linking an existing pattern_id to matching scenarios).

## Example Use Cases
- User enters 3 labeled numeric fields for a new scenario, clicks Guess. No existing pattern matches this label set. Agent searches same-labeled historical scenarios, finds a linear relationship, returns an estimate with the derived rule. User confirms it's correct. Agent stores the rule as a new pattern and links it to all scenarios sharing that label set.
- User enters a scenario whose label set has no history at all, and no abstracted pattern applies. Agent returns "I don't know." User provides the correct consequence; scenario is stored (unpatterned) for future pattern-hunting.
- A later scenario has a *different* label set than any known pattern, but its data resembles an existing abstracted pattern description (e.g., "parabolic relationship when sorted ascending"). Agent applies that pattern via the label-independent fallback. If correct, the pattern_id is linked to this one scenario only — its label set is NOT added to the pattern's known label sets.

## UI Surfaces (A2UI)
The scenario form, scorecard, and confidence slider (see original outline, Section V) remain conventional static UI — their structure never changes. A2UI is scoped specifically to the two places where the *agent's own output* is inherently variable in shape:

- **Agent Reasoning surface (Guess Phase)** — a new panel, beyond the original outline's plain "estimate" text field, where the agent shows its work: which pattern (if any) it matched, the confidence score against the user's effort-dial threshold, **which Gemini model/tier answered this particular guess** (see Effort Dial section — required on every card, not just an internal log), and — when it ran the `pattern-search` skill — a short trace of what it tried (e.g., "tested linear fit: rejected; tested sorted-ascending parabolic fit: matched, confidence 0.91"). Renders as a card via a Lit-based A2UI component; falls back to a plain "I don't know" surface when no pattern clears the threshold.
- **Pattern Captured surface (Learn Phase)** — a small confirmation panel shown after a guess is confirmed correct and a pattern is written or linked: shows the abstracted text description, the rule/code snippet, and (if a new pattern) how many existing scenarios got linked to it.

Transport: A2UI rides the official **A2A extension** (`https://a2ui.org/a2a-extension/a2ui/v0.8`), since the ADK backend already speaks A2A — no need for a separate AG-UI integration. This means the `adk_a2a` scaffold template is used instead of the plain `adk` template, even though there is only one agent (no agent-to-agent coordination) — `adk_a2a` just gives us the spec-compliant A2A server (AgentCard, DataParts, etc.) that the A2UI extension is designed to layer on top of.

Renderer: **Lit web components** — the only fully-supported official A2UI renderer today (React support is roadmap-only). The Lit surface is mounted as a custom element inside the rest of the frontend (which can otherwise be plain HTML/JS), so only the two agent-output panels need to speak A2UI.

## Effort Dial (unified slider)
One UI slider — non-technical labels (e.g., "Quick & Economical" ↔ "Thorough & Precise") — drives everything cost/quality related, rather than separate confidence and model controls. Low end favors speed and cost; high end favors correctness and thoroughness. Internally it sets, together:

- **Model tier (Gemini-only)** — "Economy" = Gemini Flash-Lite (thinking off), "Balanced" = Gemini Flash, "Max Quality" = Gemini Pro. Staying within the Gemini family keeps this a simple string swap on the ADK agent's `model` field — no extra auth, secrets, or billing accounts, and it keeps the deployment story 100% GCP-native. Exact model IDs to be confirmed at build time via the model-listing command (per `/google-agents-cli-workflow`) rather than hardcoded from training data.
- **Thinking budget** — the Gemini 2.5+ "thinking budget" API parameter, scaled with dial position (low = minimal/off, high = larger budget for the reasoning-heavy pattern-search steps).
- **Search persistence** — how many strategies `pattern-search` tries before giving up: low = script only, no freeform fallback; high = script → freeform hypotheses → more sandboxed regression attempts.
- **Acceptance threshold** — the confidence bar a candidate pattern must clear to be returned instead of "I don't know."

Every Agent Reasoning card must state which model/tier actually answered that guess (see UI Surfaces above) — since the dial can change between scenarios, the scorecard and reasoning trace need to stay honest about which "effort level" produced each result.

## Tools Required
- **Pattern Finder DB (via custom MCP server)** — Cloud SQL (Postgres) instance backing the `scenarios`, `scenario_inputs`, `labels`, `patterns`, `pattern_labels` tables (schema from `Pattern Finder Outline.txt`). Exposes tools such as: `get_pattern_by_label_set`, `get_scenarios_by_label_set`, `get_all_pattern_descriptions`, `insert_scenario`, `upsert_pattern`, `link_pattern_to_scenarios`. Both the deterministic app layer and the agent call the DB exclusively through this MCP server — no raw SQL from agent code.
- **`VertexAiCodeExecutor`** (`google.adk.code_executors`, `stateful=True`, `optimize_data_file=True`) — ADK's built-in sandboxed executor, confirmed via the `data-science` reference sample. No custom sandbox needs to be built: it runs agent-written Python in an isolated environment with common libraries pre-imported, keeps state across snippets within a search, and feeds captured stdout back to the agent as text. Used by `pattern-search` when its existing script doesn't find a match.

## Constraints & Safety Rules
- The agent may never write to the database directly — only through the validated MCP tools listed above (prevents malformed/destructive writes).
- Sandboxed code execution is isolated from the filesystem and network; it only receives the scenario data relevant to the current search and returns a result/confidence value.
- The agent must never claim a pattern match without a confidence value; below the effort dial's acceptance threshold it must return "I don't know" rather than guess.
- Pattern label-set updates only happen on an exact label-set match path (Section VII of the outline); label-independent matches must NOT cause the pattern's associated label sets to be updated — only the single matching scenario gets linked.
- When the pattern-search skill discovers a new working approach outside its existing script, the script itself should be updated (self-improving skill) — but this file edit should be scoped to the skill's own script, not other project code.

## Success Criteria
- Given a set of same-labeled scenarios with a genuine underlying rule (numeric, logical, or qualitative), the agent finds and correctly applies that rule within a small number of examples.
- The agent's "I don't know" rate and correct/incorrect rate are both visible and explainable via the UI scorecard.
- All 3 capstone key concepts are demonstrably present: Agent Skills (pattern-search, pattern-apply, pattern-abstract, label-independent-match), MCP (DB access), and a working Cloud Run deployment on GCP.

## Architecture & Sub-Agents
Single agent, invoked at two handoff points (Guess Phase, Learn Phase) — not a multi-agent system. Scaffolded with the `adk_a2a` template solely to get a spec-compliant A2A server for the A2UI extension (see UI Surfaces section) — there is no second agent. Four Skills:
- `pattern-search` — evolvable script-first, freeform-reasoning-second pattern discovery over same-labeled scenarios; uses sandboxed code execution. **Seeded from an existing script, `Desktop/pattern_finder.py`** (structural presence-pattern check, degree<=2 polynomial least-squares fit with sparse/underdetermined handling, single-variable transform battery for digit-based puzzles). This becomes the skill's v1 script and is the one the agent edits/extends over time per Section VII of the outline. One adaptation to watch for during the build phase: it's numeric-only today (`np.linalg.lstsq` over floats) — the outline allows textual/qualitative input values too, so handling those will be one of the first things the agent's self-improvement loop needs to add to the script. Its print-based output turns out to be a non-issue: `VertexAiCodeExecutor` captures stdout and feeds it back to the agent as text, which is exactly the format `analyze()` already produces — no rewrite to structured returns needed.
- `pattern-apply` — runs a stored rule/code or interprets a text-based pattern directive against new scenario data.
- `label-independent-match` — fallback: scans existing abstracted pattern descriptions when no label-based pattern applies.
- `pattern-abstract` — (Learn Phase) writes a label-free text description plus rule/code for a newly confirmed pattern.

## Data Sources & Auth
Fresh GCP project. Cloud SQL (Postgres) for the Pattern Finder DB, provisioned as part of setup. No external third-party APIs or auth beyond standard GCP service credentials.

## Non-Functional Requirements
- Deployment: prototype first (no CI/CD, no Terraform) to iterate on agent + MCP + skills logic; add Cloud Run deployment scaffolding via `scaffold enhance` once the core loop works end-to-end.
- Session storage: in-memory is sufficient — the durable state of the system lives in the Pattern Finder DB via MCP, not in agent conversation memory.
- AgentCard must declare the A2UI extension (`https://a2ui.org/a2a-extension/a2ui/v0.8`) with its supported catalog ID(s) for the Agent Reasoning and Pattern Captured surfaces.
- Frontend adds a Lit dependency for the two A2UI-rendered panels; the rest of the frontend (form, scorecard, slider) has no A2UI dependency.

## Reference Samples
Studied during Phase 1 — concrete patterns to reuse:
- `genmedia-for-commerce` (`genmedia4commerce/fast_api_app.py`) — single FastAPI app built from `google.adk.cli.fast_api.get_fast_api_app()`, with custom REST routers mounted alongside it and the frontend's static files served from the same app. `genmedia4commerce/agent.py` shows the agent connecting to its MCP server via `McpToolset` (`SseConnectionParams` for deployed, `StdioConnectionParams` for local dev), plus `before_tool_callback`/`after_tool_callback`/`before_model_callback`/`after_model_callback` hooks — Pattern Finder's A2UI DataParts (Agent Reasoning / Pattern Captured surfaces) are the natural place to hook in via `after_model_callback`, following this sample's pattern for embedding structured data in responses.
- `data-science` (`data_science/sub_agents/analytics/agent.py`) — confirms `VertexAiCodeExecutor` as the drop-in sandboxed executor (see Tools Required above); no custom sandbox needed.

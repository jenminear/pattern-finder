# Agent Spec — Pattern Finder

## Overview
A system that recognizes patterns in small, labeled, tabular scenarios (up to 5 input fields, each user-labeled), stores discovered patterns for reuse, and applies them to guess the "consequence" of new scenarios. Built for a Kaggle AI Agents capstone (Freestyle category), demonstrating three required concepts — Agent Skills, MCP for database access, deployment on Google Cloud — plus a fourth, additional concept: A2UI for the agent's own output surfaces.

The agent is invoked at two distinct handoff points from a custom web UI (not a general chat interface), as two turns of ONE ADK session (not two agents):

1. **Guess Phase** — triggered when the user clicks "Guess the Consequence" after entering a scenario. Most turns never reach the LLM at all: a deterministic pre-check (`app/agent.py`'s `_before_agent_dispatch` / `_guess_phase_content`) first tries an exact label-set + existing-pattern match, applying the stored rule directly via a safe AST evaluator when it's a plain numeric expression; if that comes up empty, it runs the seeded `pattern-search` script deterministically and applies its result if confident. Only when BOTH come up empty — or the matched pattern's rule needs judgment to interpret (a text directive, not a numeric expression) — does the turn hand off to the LLM agent, with the deterministic layer's findings appended as instruction context so the agent doesn't redo that work. See Deterministic Architecture below.
2. **Learn Phase** — triggered after the user submits the correct consequence. This phase now *always* resolves deterministically in the same dispatcher (never reaches the LLM agent's tool loop): recording the scenario is always deterministic; comparing the guess to the correct answer is deterministic for numeric answers and a narrow (non-agent) LLM call otherwise; capturing/linking a pattern is a deterministic DB write when an existing pattern was reused, and only a narrow LLM call (to write the new pattern's label-free description) when pattern-search found something new.

## Example Use Cases
- User enters 3 labeled numeric fields for a new scenario, clicks Guess. No existing pattern matches this label set. Agent searches same-labeled historical scenarios, finds a linear relationship, returns an estimate with the derived rule. User confirms it's correct. Agent stores the rule as a new pattern and links it to all scenarios sharing that label set.
- User enters a scenario whose label set has no history at all, and no abstracted pattern applies. Agent returns "I don't know." User provides the correct consequence; scenario is stored (unpatterned) for future pattern-hunting.
- A later scenario has a *different* label set than any known pattern, but its data resembles an existing abstracted pattern description (e.g., "parabolic relationship when sorted ascending"). Agent applies that pattern via the label-independent fallback. If correct, the pattern_id is linked to this one scenario only — its label set is NOT added to the pattern's known label sets.

## UI Surfaces (A2UI)
The scenario form, scorecard, and confidence slider (see original outline, Section V) remain conventional static UI — their structure never changes. A2UI is scoped specifically to the two places where the *agent's own output* is inherently variable in shape:

- **Agent Reasoning surface (Guess Phase)** — a new panel, beyond the original outline's plain "estimate" text field, where the agent shows its work: which pattern (if any) it matched, the confidence score against the user's effort-dial threshold, **which Gemini model/tier answered this particular guess** (see Effort Dial section — required on every card, not just an internal log), and — when it ran the `pattern-search` skill — a short trace of what it tried (e.g., "tested linear fit: rejected; tested sorted-ascending parabolic fit: matched, confidence 0.91"). Renders as a card via a Lit-based A2UI component; falls back to a plain "I don't know" surface when no pattern clears the threshold.
- **Pattern Captured surface (Learn Phase)** — a small confirmation panel shown after a guess is confirmed correct and a pattern is written or linked: shows the abstracted text description, the rule/code snippet, and (if a new pattern) how many existing scenarios got linked to it.

Transport: A2UI rides the official **A2A extension** (`https://a2ui.org/a2a-extension/a2ui/v0.8`), since the ADK backend already speaks A2A — no need for a separate AG-UI integration. This means the `adk_a2a` scaffold template is used instead of the plain `adk` template, even though there is only one agent (no agent-to-agent coordination) — `adk_a2a` just gives us the spec-compliant A2A server (AgentCard, DataParts, etc.) that the A2UI extension is designed to layer on top of.

Renderer: **Lit web components** — the only fully-supported official A2UI renderer today (React support is roadmap-only). The Lit surface is mounted as a custom element inside the rest of the frontend (which can otherwise be plain HTML/JS), so only the two agent-output panels need to speak A2UI.

## Effort Dial (unified slider)
One UI slider — non-technical labels (e.g., "Quick & Economical" ↔ "Thorough & Precise") — drives everything cost/quality related, rather than separate confidence and model controls. Low end favors speed and cost; high end favors correctness and thoroughness. Internally it sets, together:

- **Model tier (Gemini-only)** — "Economy" = Gemini Flash-Lite (thinking off), "Balanced" = Gemini Flash, "Max Quality" = Gemini Pro. Staying within the Gemini family keeps this a simple string swap on the ADK agent's `model` field — no extra auth, secrets, or billing accounts, and it keeps the deployment story 100% GCP-native. Exact model IDs to be confirmed at build time via the model-listing command (per `/google-agents-cli-workflow`) rather than hardcoded from training data.
- **Thinking budget** — the Gemini thinking-config API parameter, scaled with dial position (low = minimal/off, high = larger budget for the reasoning-heavy pattern-search steps). Two mutually-exclusive fields exist across the model lineup and must be matched per-model, not assumed: `gemini-2.5-pro` (Max Quality) only accepts the older `thinking_budget` (integer token count; even `0` 400s -- it can't disable thinking, so Max Quality uses `32768`, that model's max), while the `gemini-3.x` models (Economy, Balanced) accept the newer qualitative `thinking_level` (and, confirmed live, `thinking_budget` too, though the project sticks to one field per tier by convention). Confirmed empirically against the live API per model, not assumed from training data, after a real bug shipped with Max Quality set to `thinking_level="HIGH"` and 400ing on every request.
- **Search persistence** — how many strategies `pattern-search` tries before giving up: low = script only, no freeform fallback; high = script → freeform hypotheses → more sandboxed regression attempts.
- **Acceptance threshold** — the confidence bar a candidate pattern must clear to be returned instead of "I don't know."

Every Agent Reasoning card must state which model/tier actually answered that guess (see UI Surfaces above) — since the dial can change between scenarios, the scorecard and reasoning trace need to stay honest about which "effort level" produced each result. When a turn resolves deterministically (no LLM call at all), the card states that plainly instead of naming a model.

## Deterministic Architecture
Most of both Guess and Learn phases resolve without an LLM call — the LLM is invoked only where genuine judgment is needed. This was a deliberate revision from an earlier all-agent design, driven by the observation that most steps (DB lookups, exact-match rule application, running the seeded script, numeric answer comparison, copying a known pattern_id) have no judgment call to make at all.

- **Where the LLM is still used**: (a) interpreting a known pattern whose rule is a text directive rather than a numeric expression, (b) freeform pattern-search reasoning when the seeded script finds nothing, (c) comparing a non-numeric guess to the correct answer, (d) writing a newly-confirmed pattern's abstracted, label-free description. (a) and (b) go through the full `root_agent`/`pattern_search_agent` (need tools, multi-step reasoning); (c) and (d) are narrow, single-turn calls via the raw `google.genai.Client`, bypassing the agent/tool machinery entirely since there's no tool use or multi-step reasoning involved.
- **x0/x1 rule convention**: every stored numeric rule uses positional variable names (`x0`, `x1`, `x2`, ...) for the scenario's labeled inputs — never the actual label names. This keeps a rule label-independent, matching the property an abstracted pattern's text description already has, so the rule stays applicable when later reused via a label-independent match against a scenario with entirely different labels. "Order" is anchored to the labels **sorted alphabetically by name** — not the order labels happen to appear in any given request (which varies by which row the user typed each label into, since the DB stores label sets, not label order). Without this anchor, the same label set submitted in two different orders would silently reassign which value is x0/x1/... and corrupt a stored rule's meaning. `_guess_phase_content` always sorts before building `x0.../numeric_values`, and every handoff instruction to the LLM includes the explicit `x0=<label>, x1=<label>, ...` mapping rather than asking the model to re-derive the sort itself.
- **`_before_agent_dispatch`** (`before_agent_callback` on `root_agent`) is the single interception point for every turn. Returning `genai_types.Content` from a `before_agent_callback` sets `ctx.end_invocation = True` in ADK's `BaseAgent.run_async`, skipping the LLM/tool loop entirely — this is what lets the deterministic paths bypass the agent altogether rather than just short-circuiting inside its instructions.
- **Handoff instruction passing**: when the deterministic pre-check can't resolve a Guess-phase turn, it needs to tell the agent *why* it was invoked and what's already been ruled out. ADK session state written in `before_agent_callback` was empirically NOT reliable for later instruction-template building on the ADK version used here (state deltas came back empty on the yielded event and the next turn's session state, despite `has_delta()` reporting `True` inside the callback itself). Fixed with a module-level dict keyed by `invocation_id` (`_handoff_instructions`), populated in `_before_agent_dispatch` and consumed via `LlmRequest.append_instructions(...)` inside the existing `before_model_callback` (`_apply_effort_dial_and_context`) — the same mechanism already used reliably for the model/thinking-config override. An `after_agent_callback` (`_cleanup_handoff_instructions`) evicts the entry so the cache doesn't grow unboundedly.
- **EMIT_UI toggle**: the frontend's "Show agent reasoning details" checkbox sends an `EMIT_UI: on|off` line. When off, the deterministic paths still resolve the same way but omit the A2UI DataPart from the response (plain text only), and the LLM-handoff instruction tells the agent not to call `emit_agent_reasoning`/`emit_pattern_captured` either — a cheap on/off switch for the two UI panels without touching their underlying logic.

## Tools Required
- **Pattern Finder DB (via custom MCP server)** — Cloud SQL (Postgres) instance backing the `scenarios`, `scenario_inputs`, `labels`, `patterns`, `pattern_labels` tables (schema from `Pattern Finder Outline.txt`). Exposes tools such as: `get_pattern_by_label_set`, `get_scenarios_by_label_set`, `get_all_pattern_descriptions`, `insert_scenario`, `upsert_pattern`, `link_pattern_to_scenarios`. Both the deterministic app layer and the agent call the DB exclusively through this MCP server — no raw SQL from agent code.
- **`BuiltInCodeExecutor`** (`google.adk.code_executors`) — model-internal sandboxed executor. `VertexAiCodeExecutor` (the `data-science` reference sample's choice) was tried first, but its constructor makes a live network call and creates a Vertex AI Extension resource on first use — an import-time side effect and an extra cloud resource we don't need. `BuiltInCodeExecutor` gives the same isolation guarantee with no separate resource to provision and no network call at construction. Used by `pattern-search` when its existing script doesn't find a match.

## Constraints & Safety Rules
- The agent may never write to the database directly — only through the validated MCP tools listed above (prevents malformed/destructive writes).
- Sandboxed code execution is isolated from the filesystem and network; it only receives the scenario data relevant to the current search and returns a result/confidence value.
- The agent must never claim a pattern match without a confidence value; below the effort dial's acceptance threshold it must return "I don't know" rather than guess.
- Pattern label-set updates only happen on an exact label-set match path (Section VII of the outline); label-independent matches must NOT cause the pattern's associated label sets to be updated — only the single matching scenario gets linked.
- When the pattern-search skill discovers a new working approach outside its existing script, the script itself should be updated (self-improving skill) — but this file edit should be scoped to the skill's own script, not other project code.

## Success Criteria
- Given a set of same-labeled scenarios with a genuine underlying rule (numeric, logical, or qualitative), the agent finds and correctly applies that rule within a small number of examples.
- The agent's "I don't know" rate and correct/incorrect rate are both visible and explainable via the UI scorecard.
- All 3 capstone key concepts are demonstrably present: Agent Skills (pattern-search, pattern-apply, pattern-abstract, label-independent-match), MCP (DB access), and a working Cloud Run deployment on GCP.
- Verified: 21 unit tests pass with no network/LLM dependency at all -- 10 for `app/mcp_server/db_ops.py` (exact-match/safety-rule logic) and 11 for `app/agent.py`'s deterministic Guess/Learn orchestration (`tests/unit/test_agent_deterministic.py`: exact-pattern numeric apply, request-label-order robustness regression test (see below), EMIT_UI on/off, text-directive handoff, seeded-script confident fit, conservative underdetermined-history handling, Learn-phase no-match/exact-pattern/label-independent-match paths). Of 3 integration tests (real `Runner`, real MCP subprocess) against the real agent stack, 2 exercise the deterministic fast paths end to end and pass with **zero LLM calls** (`test_guess_then_learn_applies_and_captures_pattern`, `test_guess_via_seeded_script_needs_no_llm_call`); the third (`test_guess_with_no_history_says_dont_know`) genuinely needs a live model call, since there's no deterministic answer for a label set with no history and no script match. Plus 5 inherited A2A-server e2e tests (fixed a hardcoded wrong RPC path from the scaffold's generic "app" name, and a trailing-slash 405 on the RPC route) and 3 eval cases (response quality, mandated-lookup-order compliance, turn count) graded via `agents-cli eval grade`. `agents-cli eval generate` doesn't support `McpToolset` as of CLI 0.5.0 (see `tests/eval/datasets/README.md`) -- traces are generated locally via ADK's Runner instead.
- Found and fixed a second real bug (caught by user review, not automated tests): the x0/x1/... convention initially anchored "order" to whatever order labels happened to appear in the current request's text -- since the DB stores label sets, not label order, the same scenario submitted with its labels listed in a different order (e.g. entered into different form rows) would silently reassign which value is x0/x1/... and corrupt a stored rule's meaning. Fixed by anchoring x0/x1/... to the label set sorted alphabetically everywhere it's assigned (`_guess_phase_content`), with the explicit `x0=<label>, x1=<label>, ...` mapping always given to the LLM in handoff instructions rather than left for it to re-derive. Covered by `test_exact_pattern_apply_is_robust_to_request_label_order`.
- Found and fixed a real production bug along the way, not just a test artifact: the MCP subprocess's environment only inherits a curated "safe" allowlist by default (`mcp.client.stdio.get_default_environment`), not the full parent environment -- `PATTERN_FINDER_DB_URL` (and any future Cloud SQL connection vars) would have silently never reached it. Fixed in `app/agent.py`'s `db_mcp_toolset` by passing `env=dict(os.environ)` explicitly.

## Architecture & Sub-Agents
Single agent, invoked at two handoff points (Guess Phase, Learn Phase) — not a multi-agent system. Scaffolded with the `adk_a2a` template solely to get a spec-compliant A2A server for the A2UI extension (see UI Surfaces section) — there is no second agent (`pattern_search_agent` is an `AgentTool` sub-agent, not a peer). Four Skills, most now implemented as plain deterministic functions rather than agent-invoked tools (see Deterministic Architecture above):
- `pattern-search` — script-first (`app/skills/pattern_search_script.py`'s `find_pattern`/`apply_pattern`, called directly and deterministically from `_guess_phase_content` — no LLM, no sandboxed execution, no tool call), freeform-reasoning-second. The freeform fallback is `pattern_search_agent`, only ever invoked after the seeded script has already run deterministically and found nothing trustworthy — its instructions tell it not to re-derive or re-run the script, just to reason about alternative structures using its sandboxed code execution. **Seeded from an existing script, `Desktop/pattern_finder.py`** (structural presence-pattern check, degree<=2 polynomial least-squares fit with sparse/underdetermined handling, single-variable transform battery for digit-based puzzles). This is the skill's v1 script and the one a developer extends over time per Section VII of the outline (self-modifying the script file from within the running agent is out of scope for this build — see project README). One known gap: it's numeric-only today (`np.linalg.lstsq` over floats) — the outline allows textual/qualitative input values too, so handling those is a natural first extension. `find_pattern`/`apply_pattern` are deliberately conservative: an underdetermined polynomial fit (fewer data rows than free coefficients) returns `None` rather than trusting an arbitrary exact fit among infinitely many, matching the original `analyze()` function's own "UNTRUSTWORTHY" warning.
- `pattern-apply` — `evaluate_numeric_rule` (a safe, AST-restricted arithmetic evaluator, x0/x1/... convention) applies a stored numeric rule directly and deterministically whenever an exact-label-set pattern is found; only a text-directive rule (not a plain numeric expression) hands off to the LLM agent to interpret and apply.
- `label-independent-match` — fallback: scans existing abstracted pattern descriptions when no label-based pattern applies. Still an LLM-agent path (`get_all_pattern_descriptions` + judgment about which description plausibly fits), reached via the Guess-phase handoff when neither the exact-pattern nor seeded-script deterministic checks find anything.
- `pattern-abstract` — (Learn Phase) writes a label-free text description plus rule/code for a newly confirmed pattern. Deterministic (`db_ops.upsert_pattern`/`link_pattern_to_scenarios`) except for the description text itself, which is a narrow (non-agent) LLM call (`_write_pattern_description`) since that's the one genuinely creative part.

## Data Sources & Auth
Fresh GCP project. Cloud SQL (Postgres) for the Pattern Finder DB, provisioned as part of setup. No external third-party APIs or auth beyond standard GCP service credentials.

## Non-Functional Requirements
- Deployment: prototype first (no CI/CD, no Terraform) to iterate on agent + MCP + skills logic; add Cloud Run deployment scaffolding via `scaffold enhance` once the core loop works end-to-end.
- Session storage: in-memory is sufficient — the durable state of the system lives in the Pattern Finder DB via MCP, not in agent conversation memory.
- AgentCard must declare the A2UI extension (`https://a2ui.org/a2a-extension/a2ui/v0.8`) with its supported catalog ID(s) for the Agent Reasoning and Pattern Captured surfaces.
- Frontend adds a Lit dependency for the two A2UI-rendered panels; the rest of the frontend (form, scorecard, slider) has no A2UI dependency.

## Reference Samples
Studied during Phase 1 — concrete patterns to reuse:
- `genmedia-for-commerce` (`genmedia4commerce/fast_api_app.py`) — single FastAPI app built from `google.adk.cli.fast_api.get_fast_api_app()`, with custom REST routers mounted alongside it and the frontend's static files served from the same app. `genmedia4commerce/agent.py` shows the agent connecting to its MCP server via `McpToolset` (`SseConnectionParams` for deployed, `StdioConnectionParams` for local dev), plus `before_tool_callback`/`after_tool_callback`/`before_model_callback`/`after_model_callback` hooks — Pattern Finder's A2UI DataParts (Agent Reasoning / Pattern Captured surfaces) are the natural place to hook in via `after_model_callback`, following this sample's pattern for embedding structured data in responses.
- `data-science` (`data_science/sub_agents/analytics/agent.py`) — showed the sandboxed-executor + AgentTool pattern; we adapted it to `BuiltInCodeExecutor` instead of its `VertexAiCodeExecutor` choice (see Tools Required above) to avoid provisioning a separate Vertex AI Extension resource.
